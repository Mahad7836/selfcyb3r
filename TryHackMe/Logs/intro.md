# TryHackMe Notes: Intro to Logs 

This document covers the foundational concepts of what logs are, why they are critical for security operations, and how they are managed at scale within an enterprise environment.

---

## Intro to Logs

### 📌 Overview
Logs are essentially the digital footprints left behind by users, applications, and operating systems. In cybersecurity, if a system isn't logging, you are entirely blind to what is happening on your network.

### 1. What is a Log?
A log is a time-stamped record of an event that happened within a system. They are used for:
* **Troubleshooting:** Figuring out why an application crashed.
* **Incident Response (IR):** Reconstructing the timeline of a cyber attack.
* **Compliance:** Proving to auditors that security controls are functioning (e.g., HIPAA, PCI-DSS).

### 2. The Anatomy of a Log Entry
While logs come in many formats, a well-structured log typically contains:
* **Timestamp:** *When* the event occurred (usually in UTC to prevent timezone confusion).
* **Source:** *Where* the event originated (IP address, hostname).
* **User/Identity:** *Who* triggered the event (Username, Service Account).
* **Action/Event ID:** *What* happened (e.g., "Login Failed", Windows Event ID 4625).
* **Status/Outcome:** *Result* of the action (Success, Failure, Error).

### 3. Common Log Formats & Types
* **Plain Text / Syslog:** Traditional Linux/Unix logging format. Easy to read but harder to query automatically.
* **JSON (JavaScript Object Notation):** Key-value pairs. Highly structured and the preferred format for modern log ingestion tools.
* **XML / Windows Event Logs (EVTX):** Microsoft's proprietary binary logging format. Requires the Event Viewer or specific tools to read.
* **Web Server Logs:** Generated by Apache/Nginx (e.g., Access Logs showing HTTP GET/POST requests, User-Agents, and HTTP Status Codes).
